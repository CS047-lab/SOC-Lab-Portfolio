# SSH Brute Force Detection and Post-Compromise Investigation using Wazuh

## Overview

This project demonstrates how Wazuh detects an SSH brute-force attack against an Ubuntu server in a home SOC lab.

After the brute-force simulation, a successful SSH login was performed using valid credentials to simulate an attacker who has already obtained access. Once inside the system, a new local user account was created to demonstrate persistence and to observe additional security events generated by Wazuh.

---

# Lab Environment

| Component | Technology |
|-----------|------------|
| SIEM | Wazuh |
| Attacker | Kali Linux |
| Victim | Ubuntu 24.04 |
| Monitoring | Wazuh Agent |
| Attack Tool | Hydra |
| Virtualization | VMware Workstation |

---

# Objective

- Simulate an SSH brute-force attack.
- Investigate authentication alerts generated by Wazuh.
- Simulate successful SSH access using valid credentials.
- Create a new Linux user account to simulate attacker persistence.
- Analyze the complete attack lifecycle from initial access to persistence.

---

# Attack Scenario

The attack was performed in the following stages:

1. SSH brute-force attack from Kali Linux
2. Multiple failed authentication attempts
3. Wazuh generated authentication alerts
4. Successful SSH login using valid credentials
5. Creation of a new local user account
6. Investigation of all generated security events

---

# Attack Simulation

## Step 1 - SSH Brute Force

### Tool

Hydra

### Command

```bash
hydra -l <USERNAME> -P rockyou.txt ssh://<TARGET_IP>
```

Example

```bash
hydra -l student -P rockyou.txt ssh://192.168.126.129
```

This generated multiple failed SSH authentication attempts.

---

## Step 2 - Successful SSH Login

After completing the brute-force simulation, a successful SSH login was performed using valid credentials available in the lab environment.

### Command

```bash
ssh student@192.168.126.129
```

### Verification

```bash
whoami
hostname
pwd
```

Example Output

```text
student
ubuntu
/home/student
```

This represents an attacker obtaining interactive shell access after acquiring valid credentials.

---

## Step 3 - Persistence Simulation

After gaining shell access, a new local user account was created.

### Command

```bash
sudo adduser attacker
```

Verify user creation

```bash
cat /etc/passwd | grep attacker
```

Example Output

```text
attacker:x:1002:1002::/home/attacker:/bin/bash
```

Creating an additional account simulates an attacker attempting to maintain persistent access.

---

# Detection

Wazuh successfully collected authentication logs from Ubuntu and generated alerts for repeated failed SSH login attempts.

The user creation event was also logged by the operating system and collected for investigation.

# Indicators of Compromise (IOCs)

- Multiple failed SSH authentication attempts
- High volume of login failures from a single source IP
- Successful SSH login
- Creation of a new local user account
- Changes to the `/etc/passwd` file

---

# Analysis

The attack began with repeated SSH authentication attempts using Hydra. Ubuntu recorded each failed login attempt in the authentication logs, which were forwarded to the Wazuh Manager by the Wazuh Agent.

After the brute-force simulation, a successful SSH login was performed using valid credentials to represent an attacker who had obtained legitimate access. Once authenticated, a new Linux user account was created to simulate persistence. Wazuh collected both the authentication activity and the account creation events, providing visibility into the attack lifecycle from initial access through post-compromise activity.

---

# Remediation

- Enable Multi-Factor Authentication (MFA)
- Use SSH key-based authentication
- Disable password authentication where possible
- Disable direct root SSH login
- Implement Fail2Ban
- Enforce strong password policies
- Monitor privileged account creation
- Regularly review authentication logs
- Audit newly created user accounts
